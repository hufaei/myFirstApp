# Delivery Strategy

## 1. 目标

LifeLab 不按“大一统全量实现”推进，而按 `Modular Monolith + Vertical Slice Delivery` 推进。

这意味着：

- 架构边界先在单 `app` module 内建立
- 每次只实现一条具备运行入口、可测试、可静态验证的垂直切片
- 每个切片进入开发前，都要单独完成设计文档和 implementation plan

## 2. 为什么采用切片交付

如果直接按“全 app 总蓝图”实施，会有三个问题：

- 反馈周期过长，架构问题要到很后面才暴露
- 文档会不可避免地写出大量没有被验证的细节
- 实施计划会过大，难以保持高质量和高一致性

切片交付的好处是：

- 先用第一条完整链路验证核心架构
- 文档只在真正需要时细化
- 每个阶段都能独立验收，不靠主观感觉判断“差不多做完”

## 3. 切片顺序

### Slice A: Platform Baseline

目标：

- 将当前 Kotlin JVM 模板迁移为 Android 工程
- 建立 Compose 宿主、导航骨架、Hilt、基础 Result/ErrorModel、主题和测试基线
- 只建立当前切片需要的最小基础设施，不提前做完整网络、数据库或设置体系

包含：

- `Application`
- `MainActivity`
- 底部导航壳或主入口壳
- 核心依赖注入
- 统一错误和结果模型
- 最小假数据与环境开关
- 最小 UI smoke 验证与单元测试基线

退出条件：

- Android 工程可以构建
- App 具备明确的启动入口和导航骨架
- 基础依赖注入可工作
- 当前切片相关测试任务可执行并通过
- 本机静态验证通过；设备或模拟器运行由用户在具备完整环境的机器上执行

### Slice B: Productivity Slice

目标：

- 优先打通 `Tasks + Habits` 这条最有学习价值的核心业务链路

包含：

- 任务列表、详情、新增/编辑、完成/恢复、筛选
- 习惯列表、打卡、连续记录、提醒设置
- Room、DataStore、Repository、UseCase、通知/提醒主链路

退出条件：

- 任务与习惯主要业务流具备运行入口和自动化验证路径
- 首批 UseCase、Repository、ViewModel 测试齐备
- 至少一条本地数据持久化链路跑通

### Slice C: Content Discovery Slice

目标：

- 扩展内容流能力，验证列表混排、图片、分页和搜索体验

包含：

- 首页推荐流
- 发现页内容/商品混排
- 搜索历史、搜索结果、筛选

退出条件：

- 推荐流和搜索链路具备运行入口和自动化验证路径
- 图片加载、分页、空态和错误态行为稳定

### Slice D: Account And Notifications Slice

目标：

- 完成账号、偏好和消息通知相关能力

包含：

- 个人信息
- 设置与偏好
- 站内消息
- 系统通知整合

退出条件：

- 偏好持久化稳定
- 消息中心与通知链路闭环
- 账号态切换行为可验证

## 4. 切片规则

- 当前切片未验收前，不进入下一切片
- 只有在出现第二个明确消费者后，才把特定能力上提到 `core`
- 切片设计只覆盖当前切片，不为远期功能写伪精确实现细节
- 每个切片都必须产出自己的 spec、plan 和验证结果
- 当前开发机器默认只要求测试、构建和静态验证；人工运行调试不作为本机阻塞条件
