# Coding Standards

## 1. 总体原则

- 代码首先为可维护性服务，其次才是“写得少”
- 默认按正式项目标准写，而不是教程式样板代码
- 边界清晰优先于局部偷懒
- 先让代码容易测试，再让它方便扩展

## 2. Kotlin 规范

### 变量与可变性

- 默认使用 `val`
- 只有确实需要重赋值时才用 `var`
- 避免跨作用域共享可变状态

### 空安全

- 优先设计为非空类型
- 禁止把 `!!` 当作常规手段
- 需要缺省值时优先用 `?:`

### 类型与建模

- 数据对象优先用 `data class`
- 有限状态优先用 `sealed class` 或 `sealed interface`
- 工具单例优先用 `object`
- 公共扩展函数只放真正有通用价值的逻辑

### 函数设计

- 函数保持短小、单一职责
- 优先使用命名参数与默认参数替代重载泛滥
- 带副作用的函数命名必须显式

## 3. Compose 规范

- 页面拆成 `Route` 和 `Screen` 两层更清晰
- `Route` 负责拿状态和事件绑定
- `Screen` 负责纯渲染
- 不在 Composable 里直接发网络请求
- 不把复杂业务逻辑塞进 UI

## 4. ViewModel 规范

- ViewModel 只暴露稳定 `UiState`
- 一次性事件使用 `UiEffect`
- 不把 Retrofit DTO 直接暴露给 UI
- 涉及业务规则编排时优先调用 use case

## 5. 分层规范

- UI 不直接访问数据库或网络
- Repository 是数据访问的唯一统一入口
- Mapper 不做业务决策
- Domain 不依赖页面层
- 只有在出现第二个真实消费者后，能力才上提到 `core`
- 禁止为未来假想需求预建空抽象

## 6. 命名规范

- 包名全小写
- 类名 `UpperCamelCase`
- 函数、变量 `lowerCamelCase`
- 常量 `UPPER_SNAKE_CASE`
- 布尔值命名用 `is/has/can/should`

## 7. 文件组织

- 一个文件只承载强相关内容
- 复杂文件优先拆分，而不是继续堆
- 不建立“万能 utils 文件”
- 一个 feature 的实现优先在自己的垂直切片内闭环

## 8. 测试规范

- 测试命名应体现行为和条件
- 关键业务规则必须单元测试覆盖
- UI 测试关注状态与交互，不关注像素级细节
- 使用 fake 或 stub 时，保持行为可预测

## 9. 注释规范

- 优先让代码自解释
- 只在边界条件、复杂规则、非直观决策处加注释
- 禁止写“把值赋给变量”这类无信息注释
